## 1. One-line threat
If a request can be forged, the server cannot prove "did this really come from that caller."

## 2. Concept
A token is a badge, a key is a stamp, a signature is the sealed wax over the message. They prove _different things_.

## 3. Vulnerable code
```ts
// Trusting the token alone — a changed body goes unnoticed
if (req.headers["x-token"] === SECRET) accept(req);
```

## 4. How the attack works
A token proves "who." It says nothing about body integrity, so a man-in-the-middle can rewrite the payload and still pass.

## 5. The fix
```ts
// HMAC over the body — if the body changes, the signature breaks
const sig = hmacSHA256(SECRET, rawBody);
if (timingSafeEqual(sig, req.headers["x-signature"])) accept(req);
```

| Decision | Why | Rejected |
|---|---|---|
| HMAC over the body | Proves identity AND integrity at once | Token alone: defenseless against body tampering |
| Constant-time compare | Closes timing leak | `===`: compare time leaks information |

## 6. Checklist
- [ ] **access**: does the signature cover the entire body?
- [ ] Is the comparison constant-time?
- [ ] Are secrets in env / KMS?
