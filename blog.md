# 블로그 운영·작성 원칙서 (Blog Playbook)
#spec #진행중

> 적용 범위: `Samchil.dev/blog`의 **모든** 글과 외부 재게재.
> **이 문서 하나로 글 1편을 처음부터 끝까지 쓸 수 있어야 한다.**
> 브랜드 SSOT → [[Samchil_Blueprint]] · 문서 톤 기준 → [[docs/specs/module-standard]](Detail 원칙)

---

## 0. 역할 — 이 블로그는 무엇인가

블로그는 두 가지를 동시에 한다.

1. **SEO 원본 + 신뢰 축적 레이어** (Blueprint·로드맵 Phase 2). 제품을 직접 팔지 않고, "이 회사는 보안을 안다"를 글의 밀도로 증명한다.
2. **저자의 보안 커리큘럼**. 쉬운 개념 → 어려운 개념을 한 단계씩 정리하며 토픽 권위(topical authority)를 쌓는다.

> 핵심: **장기 게재로 트래픽을 유지**하는 게 목표다. 한 방의 바이럴이 아니라, 사다리를 채워 검색 유입의 영구 자산을 만든다.

---

## 1. 정체성·포지셔닝 (확정)

| 항목 | 결정 |
|---|---|
| 저자 포지션 | **전문가 권위 + 커리큘럼** — "내가 배운다"가 아니라 "독자가 안전하게 배우도록 내가 엄격히 정리한다" |
| 원본 언어 | **한영 병행(이중 게재)** — 한 글을 두 출력으로 |
| 발행 주기 | **주 1편**(1개의 사고 = 2개 언어 출력. §5 파이프라인으로 지속) |
| 톤 | `/docs`의 **Detail 원칙** 그대로. What이 아니라 Why·Rejected까지 |

### 1-1. 단 하나의 금지선
보안 제품 회사가 "아직 배우는 중"으로 읽히면 신뢰가 깎인다. 따라서:
- **"공부 일기·초심자 고백" 프레이밍 금지.** 난이도가 낮은 글이라도 글의 *깊이*는 낮추지 않는다.
- 난이도는 **주제 선택**으로 표현하고(L1부터 시작), **사고의 밀도**는 항상 최고로 유지한다.
- "기초"는 사과가 아니라 **커리큘럼의 1교시**다. 사다리 전체를 처음부터 보여줘 의도된 설계임을 드러낸다.

## 2. 난이도 사다리 = 등급 사다리 (콘텐츠 뼈대)

글마다 **레벨(L1~L4)** 과 **4축(접근통제·입력방어·안전실패·추적책임)** 을 단다.
사다리가 제품 등급 사다리와 정렬되어, 독자가 올라올수록 제품 등급 사고에 도달한다.

| 레벨 | 성격 | 예시 주제 | 정렬 등급 |
|---|---|---|---|
| **L1 기초** | 개념·"왜 위험한가" | XSS/CSRF란, 해시 vs 암호화, 토큰이란 무엇인가 | 입문 유입 |
| **L2 적용** | 올바른 구현·흔한 실수 | HMAC 서명 바르게 쓰기, 입력 스키마 검증, 멱등성 키 | FREE / HARDENED |
| **L3 심화** | 공격·방어 심층 | 타이밍 공격, Replay 미묘함, 레이트리밋 우회 | HARDENED |
| **L4 증명** | 감사·책임·증명 | 감사 추적 설계, 논스 보장, CVE 대응 운영 | AUDITED |

- **시리즈(series) 태그**로 묶어 사다리를 시각화하고 내부 링크로 회귀 트래픽을 만든다.
- 4축 용어는 `/docs`·`/products`·[[Samchil_Grade_Criteria]]와 **반드시 동일**하게 쓴다. 사이트 전체가 한 목소리.

## 3. 글 1편 고정 템플릿 (7단계, 순서 변경 금지)

브랜드 "Proof, not promises" + Detail 원칙을 글 단위로 강제한다.

```
1. 한 줄 위협      이 글이 막는 공격 한 문장 (Hook)
2. 개념            쉽게, 비유 1개 — 비전공자도 이해
3. 취약한 코드      "이렇게 짜면 뚫린다" (실제 동작 코드)
4. 공격 원리        왜/어떻게 뚫리는가 — 시연 또는 단계 분해
5. 방어 구현        올바른 코드 + Decision / Why / Rejected 표
6. 실전 체크리스트   해당 4축 매핑 + 복붙 가능한 점검 항목
7. (soft CTA)      관련 모듈 1줄 링크 — 있을 때만, 광고 톤 금지
```

- **5번의 Decision/Why/Rejected 표가 글의 심장**이다. `/docs`와 동일 포맷.
- 코드는 한·영 버전에서 **완전히 동일**(주석만 언어별). 검증된 동작 코드만.

## 4. 메타데이터 (MDX frontmatter 규격)

모든 글은 아래 frontmatter로 시작한다. (필드명 고정)

```yaml
---
title:        # 위협/개념어 포함, 60자 이내
slug:         # 영문 케밥, 한·영 공통 키
lang:         # ko | en
level:        # L1 | L2 | L3 | L4
axis:         # [access | input | failsafe | trace] 중 1~2개
series:       # 시리즈 슬러그 (없으면 생략)
description:  # 검색 스니펫, 1 롱테일 키워드 포함, 155자 이내
date:         # YYYY-MM-DD
updated:      # 갱신 시 (CVE/패치로 본문 바뀌면 필수)
canonical:    # 자기 언어 원본 URL
hreflang:     # 짝 언어 글 URL
module:       # 연계 모듈 slug (없으면 생략)
---
```

## 5. 한영 병행 운영 — "1 craft, 2 outputs" 파이프라인

> 주 1편 × 2언어를 **따로 쓰면 번아웃**난다. 두 편이 아니라 **하나의 사고를 두 언어로 출력**한다.

```
① 국문 초안   ← 학습·사고가 일어나는 곳. 코드·Decision/Why/Rejected 완성
② 품질 게이트  ← Why·Rejected 없으면 발행 금지(§6). 국문에서 먼저 통과
③ 영문 적응   ← 직역 아님. 코드 동일, 표현만 영어답게. (AI 보조 가능)
④ 동시 발행   ← 두 페이지 + hreflang 짝 연결
⑤ 외부 요약   ← KR→Velog/Tistory, EN→Medium. 각 canonical은 자기 언어 원본
```

### 5-1. SEO 처리
- 두 언어 페이지는 서로 **canonical이 아니라 `hreflang` 짝**이다(각자 자기 언어의 원본).
- 외부 재게재(요약)만 **canonical을 사이트 원본으로** 지정한다.
- URL: 영문 `/blog/[slug]`, 국문 `/blog/ko/[slug]` (slug 공통). 동일 글의 두 언어는 slug로 짝을 찾는다.

## 6. 품질 게이트 (발행 전 필수 통과)

하나라도 미달이면 **발행하지 않는다.**

- [ ] 5번에 **Decision/Why/Rejected** 표가 있다 (What만 있으면 탈락)
- [ ] 3·5번 코드가 **실제로 동작·검증**됐다
- [ ] level·axis가 frontmatter에 있고 4축 용어가 사이트와 일치
- [ ] description에 **노리는 롱테일 키워드 1개** 포함
- [ ] "공부 일기" 톤이 아니다(§1-1)
- [ ] 외부 사실·CVE 인용 시 출처 링크가 있다

## 8. 첫 8주 시드 플랜 (사다리 바닥부터)

L1에서 시작해 의도된 커리큘럼임을 보여준다. (주제는 조정 가능, 레벨 순서는 유지)

```
W1 L1  토큰/키/서명이란 무엇인가 — 접근통제의 출발
W2 L1  해시 vs 암호화, 그리고 HMAC은 왜 다른가
W3 L2  HMAC 서명을 "바르게" 쓰는 법 (취약 코드 → 방어)        ↔ 모듈 연계
W4 L2  입력 검증: 스키마가 막는 것과 못 막는 것
W5 L2  멱등성 — 같은 요청을 두 번 받아도 안전하려면
W6 L3  타임스탬프와 논스: Replay 공격을 어떻게 끊는가
W7 L3  레이트리밋의 함정 — 우회되는 5가지 패턴
W8 L4  감사 추적은 로그가 아니다 — 책임을 증명하는 설계
```

## 9. 통합 아키텍처 — 콘텐츠 팩토리 (확정)

블로그는 **별도 레포에서 자동화 라인으로 운영**하되, **렌더링은 본 사이트가 전담**한다.
→ [[20260627-blog-integration]]

### 9-1. 도메인 비협상 원칙
- **`samchil.dev/blog` 서브경로 고정.** 서브도메인(`blog.samchil.dev`)·별도 도메인 **금지** — 검색엔진이 별개 사이트로 취급해 토픽 권위가 분산된다. 레포가 분리돼도 URL은 단일 오리진.

### 9-2. 역할 분리
```
[blog 레포 = 콘텐츠 팩토리]
  - MDX 글(ko/en) + 자동화 스크립트(생성 · 한↔영 적응 · frontmatter 검증 · §6 품질게이트 CI)
  - 렌더링·테마·라우팅 없음. 산출물은 "콘텐츠"이지 "배포된 사이트"가 아니다.
        │ push
        ▼
[본 사이트 = 렌더러]
  - blog 레포를 git submodule `content/`(또는 의존성)로 소비
  - /blog 라우트가 MDX를 렌더 · 레벨/축/시리즈 필터 · hreflang/canonical/JSON-LD 생성
        │
        ▼
  samchil.dev/blog  (단일 오리진)
```

### 9-3. 발행 트리거
- 자동화가 blog 레포에 push → blog 레포 GitHub Action이 본 사이트 **Vercel Deploy Hook** 호출 → 리빌드 시 새 글 반영.
- 즉시 반영이 필요하면 본 사이트에서 **ISR + on-demand revalidate**(`revalidatePath('/blog')`) 웹훅 사용.

### 9-4. 본 사이트 렌더 메모 (착수 시)
- `/blog`는 **MDX** 라우트. `lib/products.ts`처럼 글 인덱스 SSOT(`lib/posts.ts` 또는 contentlayer류) 1개가 `content/` 소비.
- 테마: 본문 가독성 우선 → 라이트 테마 재사용(`/docs`·`/grades`와 동일 `.theme-light`).
- 신규 CSS는 토큰 소비만(매직넘버 금지). 코드 블록은 기존 `.code-block` 재사용.
- 글 목록은 **레벨/축/시리즈 필터** 제공(사다리 시각화). 상세는 `DocsToc`식 스티키 목차 재사용 검토.
- frontmatter→hreflang/canonical/JSON-LD(Article) 자동 생성. **MDX는 빌드타임 처리**(런타임 fetch보다 커스텀 컴포넌트·코드 하이라이트에 유리).
